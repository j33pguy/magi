# Git-Backed Memory — Honest Architecture

MAGI includes an optional git-backed history layer that mirrors memory writes into a local Git repository. It is **not enabled by default** and it is **best-effort** (DB writes succeed even if Git writes fail).

## What Exists Today

The implementation lives in `internal/vcs/` and is wired into the server startup path in `internal/server/server.go`.

- `vcs.Repo` wraps `go-git` and manages a local repository, commit history, and diffs.
- `vcs.VersionedStore` wraps a concrete `*db.Client` and writes memory/link JSON files to Git on mutations.
- The server initializes Git when `MAGI_GIT_ENABLED=true` and wraps the store if the backend supports it.

### Supported Backends

Git backing is only applied when the store is a concrete `*db.Client`:

- Supported today: **SQLite** and **Turso** (they return `*db.Client`).
- Not supported today: **PostgreSQL**, **MySQL/MariaDB**, **SQL Server** (the server logs a warning and continues without Git wrapping).

## Enabling Git Backing

Set these environment variables:

- `MAGI_GIT_ENABLED=true`
- `MAGI_GIT_PATH=~/.magi/memories` (optional, defaults to this path)
- `MAGI_GIT_COMMIT_MODE=immediate` or `batch`
- `MAGI_GIT_BATCH_INTERVAL=30` (seconds, only used in batch mode)

When enabled, MAGI initializes the repo and performs an initial commit for `.magi-meta/version` if the repository does not yet exist.

## What Gets Written

Git-backed storage mirrors a subset of memory data as JSON files:

- `memories/<id>.json` per memory
- `links/<from_id>.json` per memory’s outbound links
- `.magi-meta/version` for repo metadata

Memory JSON intentionally **excludes embeddings** to keep files readable. The serialized fields mirror `SerializableMemory` in `internal/vcs/serialize.go`.

Commit messages are short and semantic, e.g.

- `remember: <content…>` on save
- `update: <content…>` on update
- `archive: <id>` on archive
- `delete: <id>` on delete
- `tags: update tags on <id>` on tag changes
- `link: update links from <id>` on link changes

In batch mode, writes are staged and committed periodically with the message `batch: update memories`.

## History And Diff APIs

When Git backing is enabled, MAGI exposes history and diff endpoints through the HTTP and gRPC servers. If Git is disabled, these endpoints return a `404`-style error response (“git versioning is not enabled”).

## Rebuild Behavior (Git → DB)

On startup, if Git is enabled **and** the database is empty **and** the repo has memory files, MAGI will rebuild the DB from Git:

- Reads each `memories/*.json` file.
- Regenerates embeddings using the active embedder.
- Saves memories into the DB and re-applies tags.

Limitations of rebuild:

- Memory IDs are **not preserved**; new IDs are generated by the database.
- Link files are **not** restored during rebuild.

## Limitations (Current State)

- Git writes are best-effort and do not block DB writes.
- Only SQLite/Turso support Git backing today.
- `DeleteLink` does not update Git history; link files are refreshed on the next `CreateLink` for that memory.
- There is no Git push/pull or remote sync built into MAGI.

## Planned (Not Implemented Yet)

- Preserve original IDs during rebuild.
- Restore link files during rebuild.
- Enable Git backing for non-SQLite backends once a compatible wrapping strategy exists.
